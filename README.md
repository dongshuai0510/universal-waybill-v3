# 运单全流程管理系统 V3

> 录单 → **扫描品控 → 异常上报 → 分级审批 → 执行联动** —— 运单全生命周期管理
>
> 承接 V2（AI 录单解析）的输出数据，**独立部署、独立数据库**，
> 通过 HTTP 接口服务与 V2 数据互通，**不直接连接 V2 数据库**。

技术栈：**Next.js 15 App Router + TypeScript**，数据库 **SQLite（本地）/ Neon Postgres（部署）**，UI 延续鲸天/V2 清爽蓝绿设计语言（主色 `#0fc6c2`）。

## 三、架构：两个独立系统，通过接口互通

```
┌────────────────────────┐        HTTP + Bearer 鉴权        ┌────────────────────────┐
│  V2 万能导入（已有）      │◀────── /api/v3/waybill ─────────│  V3 运单全流程管理（本系统）│
│  独立 Neon 库           │◀────── /api/v3/sku-check ───────│  独立 Neon 库            │
│  运单主数据             │◀────── /api/v3/waybills ────────│  工单/审批/赔付/库存/扫描  │
│                        │──────▶ /api/v3/waybill/flag ────▶│  运单只读快照 + 同步日志   │
└────────────────────────┘        (异常标记回写，可选)        └────────────────────────┘
```

- V3 **不持有运单主数据**，只在 `waybill_snapshot` 存一份**只读快照**（缓存），
  运单状态以 V2 为准。
- 发起异常上报 / 扫描录入等**关键动作实时调用 V2 校验**，不能仅凭本地快照。
- 每次调用 V2 都生成 **Request ID** 并写入 `sync_log`，可在「接口监控」页按链路追踪。
- V2 不可用时**降级**到本地快照并标注「数据同步于 XX 时间」，不白屏。

## 四、数据模型（V3 自有，11 张表）

| 表 | 说明 |
|---|---|
| `waybill_snapshot` | 运单本地只读快照（同步自 V2，禁改运单状态） |
| `sync_log` | 接口同步日志（Request ID / 接口名 / 耗时 / 状态码 / 错误码） |
| `tickets` | 异常工单（**工单状态机**，含乐观锁 version、重提计数、deadline） |
| `approval_records` | 审批记录（审计日志 + 幂等 op_token 唯一约束） |
| `payout_records` | 赔付记录（含 **direction 赔付方向**，关联 approval_record_id 可追溯） |
| `inventory` | 库存（品控暂扣 = 批次锁定 `locked` + `locked_by_ticket`） |
| `inventory_ledger` | 库存流水（关联 approval_record_id 可追溯） |
| `scan_records` | 扫描记录（**扫描批次状态机**，与工单 1:N，通过 ticket_id 关联） |
| `qc_rules` | 品控规则（可配置触发条件，非硬编码） |
| `config` | 全局配置（审批阈值/超时/重提上限，后台可调） |
| `users` | 角色与账号（含禁用状态，用于审批人兜底） |

**两套状态机分离**：`tickets.status`（工单）与 `scan_records.batch_lock_status`（批次），
通过 `scan_records.ticket_id` 关联，工单关闭前批次不自动解锁，执行动作与批次解锁在同一事务内。

## 五、核心考点落地对照

| 考点 | 落地位置 |
|---|---|
| 状态机完整性（分级/拒绝重提/超时/品控暂扣/快速放行） | `src/lib/store.ts`、`scan.ts`、`timeout-sweeper.ts` |
| 分级规则可配置 | `src/lib/config.ts`（阈值存 config 表）+ 「规则配置」页 |
| 并发冲突 | 乐观锁 `version`：`UPDATE ... WHERE id=? AND version=?`，0 行即冲突 |
| 审批人兜底 | 禁用账号 `getUserRoles` 返回空 → 拒绝；`/reassign` 转交 |
| 权限边界 | 后端校验：上报人≠审批人、层级匹配、品控主管专属快速放行 |
| 幂等性 | `approval_records.op_token` 唯一约束 + 状态前置校验 |
| 多表一致性 | `store.tx()` 单事务：状态+库存+赔付要么全成要么全回滚 |
| 可追溯 | `payout_records` / `inventory_ledger` 保留 `approval_record_id` 外键 |
| 赔付方向 | `payout_records.direction`：物流=to_customer / 品控=to_supplier_claim |
| 跨系统接口 | `src/lib/v2-client.ts`（鉴权/超时/重试/日志/降级）|
| 接口可调试性 | Request ID 贯穿 → `sync_log` → 「接口监控」页还原调用链 |
| 品控规则引擎 | `src/lib/qc-engine.ts`（规则表驱动，记录命中规则 + 判定依据）|
| AI 建议 | `src/lib/ai.ts`（需人工确认、说明依据、失败回退，不阻塞）|

## 六、本地运行

```bash
npm install
cp .env.example .env      # 可留空：不配 DATABASE_URL 走本地 SQLite；不配 V2 走降级
npm run dev               # http://localhost:3000
npm run seed              # 生成 ≥200 条演示工单（也可在页面点按钮）
```

未设置 `DATABASE_URL` 时自动使用本地 SQLite（`./data/v3.db`），零配置即可跑通。

### 环境变量

| 变量 | 说明 |
|---|---|
| `DATABASE_URL` | V3 **独立** Neon 实例（部署必填；本地留空走 SQLite） |
| `V2_API_BASE` | V2 接口基址（默认已指向线上 V2） |
| `V2_API_KEY` | 调用 V2 的 Bearer Key（须与 V2 侧 `V3_API_KEY` 一致） |
| `V2_TIMEOUT_MS` / `V2_MAX_RETRY` | 超时（默认 8s）/ 重试（默认 2） |
| `ANTHROPIC_API_KEY` | 可选：启用 LLM 分类；不配则用本地启发式 |

## 七、反思题（考点 9，不计分）

**1. 若题目直接给出精确阈值和超时，架构会怎么变？**
不会把它们写进代码常量，仍会放进 `config` 表 —— 因为业务数值天然易变（大促期、不同仓）。
给定精确值只是省去我「设定依据」的论证，配置化的架构不变。真正的差别是：给定值时我会
额外做「阈值变更审计」，因为有了明确基线，任何偏离都值得追溯。

**2. 「审批状态变更」与「库存/赔付联动」一致性的最大风险点？不用事务怎么办？**
最大风险是**部分成功**：工单已置 done，但赔付/库存未写入 → 断链、对不上账。
我用单数据库事务 `store.tx()` 消除中间态。若不能用事务（如跨库/跨服务），我会用
**Outbox + 幂等消费**：状态变更与「待联动事件」写在同一本地事务，后台任务读 outbox
执行联动，每个事件带唯一 ID 保证幂等重试，失败可重放，最终一致。

**3. 模糊需求：先问清楚还是先做一版？**
先做一版可运行的、把假设**显式写下来**（就是本项目的假设说明文档），再拿着它去对齐。
纯口头问容易发散，一个带明确假设的原型能把「产品到底要什么」逼出来，反馈质量高得多。
但对**不可逆/高成本**的决策（如赔付方向、对账口径）我会先问，因为改起来代价大。

**4. 新增第六种异常「客户投诉服务态度」（不涉及货品/库存）需要改哪些地方？改动大吗？**
改动小。① `types.ts` 加一个异常类型；② `suggestAction` 映射里给它一条
「无库存联动、无赔付 或 仅安抚补偿」的分支；③ 前端 label 加一项。
状态机、审批引擎、表结构**都不用动** —— 因为库存/赔付联动本就是「按需触发」
（`needInventory`/`direction` 可为空），执行阶段跳过即可。这正是当初把「是否联动」
做成数据驱动而非写死的收益。

**5. 工单从 200 涨到 20 万，最先撑不住的是哪个环节？怎么处理？**
最先是**定时超时任务的全表扫描**和**列表的 COUNT 分页**。
- 超时任务：已建 `idx_tickets_deadline`，改为「只扫未关闭 + deadline 已过」的索引范围查询，
  并分批处理；进一步可按 deadline 分片、用队列而非全表轮询。
- 列表：`COUNT(*)` 大表慢，改用**近似计数 / 游标分页（keyset）**替代 offset 深分页；
  筛选字段都已有索引。
- 接口同步：改**增量同步 + 队列削峰**，避免快照刷新积压。

**6. 给 V2 新增对外接口，如何不破坏现有调用方？金额字段 int→decimal 怎么办？**
- 新增接口走**独立版本命名空间**（`/api/v3/*`），不碰 V2 既有 `/api/orders`，
  纯增量、对老调用方零影响；字段只增不改不删（向后兼容）。
- `int→decimal`：V3 侧解析统一用 `Number()`（JS 无 int/decimal 之分，天然兼容数值放大）；
  真正的风险是**精度**——金额一律以数值解析后按分/两位小数处理，不用 `parseInt`。
  接口返回加 `schemaVersion`，V3 据此感知字段语义升级；灰度期两种类型并存时按 `typeof` 兼容。
