# 系统间接口文档 —— V3 调用 V2

> V3（运单全流程管理系统）**不直接连接 V2 数据库**，所有运单数据交互通过 V2 暴露的
> HTTP 接口完成。本文档说明接口列表、入参出参、鉴权、超时与重试策略、V2 不可用时的降级方案，
> 以及在存量系统 V2 上**新增对外接口的工程判断**（考点 5「老系统二开意识」）。

---

## 一、架构与数据流

```
┌────────────────────┐     HTTP + Bearer Token     ┌────────────────────┐
│  V3 运单全流程管理   │ ──────────────────────────▶ │  V2 万能导入（存量） │
│  独立部署 / 独立库   │ ◀────────────────────────── │  独立部署 / 独立库   │
└─────────┬──────────┘     JSON + x-request-id      └────────────────────┘
          │
          ▼ 只读缓存
   waybill_snapshot（本地快照，禁止改运单状态，以 V2 为准）
   sync_log（每次调用的链路追踪，含 Request ID）
```

- **生产地址**：`V2_API_BASE = https://universal-import-v2-ten.vercel.app`
- V3 在 Vercel 上以 server-to-server 方式调用 V2（同为 Vercel 出口，网络可达）。
- 所有接口位于 V2 的 `/api/v3/*` 命名空间（**版本化隔离**，不影响 V2 既有 `/api/orders` 等调用方）。

---

## 二、鉴权

- 方式：`Authorization: Bearer <V3_API_KEY>`（亦支持 `x-api-key: <key>`）。
- V2 侧通过环境变量 `V3_API_KEY` 校验；V3 侧通过 `V2_API_KEY` 持有同一密钥。
- 鉴权失败返回 **401** + 明确错误码：
  - `MISSING_CREDENTIALS`：未携带凭证。
  - `INVALID_CREDENTIALS`：凭证无效。
- **不是裸奔的开放接口**（考点 5 要求）。

## 三、接口列表

### 1. 校验运单是否存在 + 获取详情

```
GET /api/v3/waybill?code=<外部编码>
Authorization: Bearer <key>
x-request-id: <可选，V3 生成的链路 ID>
```

**成功 200**：
```json
{
  "exists": true,
  "requestId": "v3-xxx",
  "waybill": {
    "externalCode": "PS2512220005001",
    "receiverStore": "海口龙湖天街店",
    "receiverName": null, "receiverPhone": null, "receiverAddress": null,
    "totalQuantity": 120, "skuCount": 8,
    "skus": [{ "skuCode": "SKU-1001", "skuName": "商品A", "quantity": 10, "spec": "标准" }],
    "sourceFile": "xxx.xlsx", "createdAt": "2025-..."
  },
  "openExceptionFlag": { "hasOpenException": false, "ticketId": null }
}
```

**运单不存在 404**：`{ "error": "WAYBILL_NOT_FOUND", "exists": false, "message": "运单不存在：xxx", "requestId": "..." }`

> **用途**：发起异常上报时的**实时真实性校验**。V3 的 `/api/report` 强制调用它，
> 404 时拒绝上报，杜绝对不存在运单发起异常。

### 2. 校验 SKU 是否归属于指定运单

```
POST /api/v3/sku-check
Authorization: Bearer <key>
Body: { "code": "<运单外部编码>", "skuCode": "<SKU 编码>" }
```

**成功 200**：
```json
{ "belongs": true, "requestId": "v3-xxx",
  "matchedSku": { "skuCode": "SKU-1001", "skuName": "商品A", "quantity": 10 } }
```
不归属时 `belongs: false, matchedSku: null`；运单不存在返回 404。

> **用途**：扫描录入时验证该 SKU 确实在该运单 SKU 明细中，避免扫到无关货物。
> V3 的 `/api/scan` 强制先调用它，不归属返回 422 拒绝扫描。

### 3. 按条件查询/同步运单列表

```
GET /api/v3/waybills?page=1&pageSize=100&code=<可选模糊>
Authorization: Bearer <key>
```
返回 `{ waybills: [...], total, page, pageSize }`。

> **用途**：本地快照表初始化 / 增量同步（`/api/sync`）。

### 4.（加分项）异常标记回写 V2

```
POST /api/v3/waybill/flag
Authorization: Bearer <key>
Body: { "code": "<运单号>", "hasOpenException": true, "ticketId": "tl_xxx", "note": "物流异常：丢件" }
```
返回 `{ ok: true, requestId }`。V2 侧将标记存入 `waybill_exception_flags` 表，
`GET /api/v3/waybill` 会带出 `openExceptionFlag`，避免 V2 对存在未关闭异常的运单继续按正常运单处理（如重复发货）。

> V3 在上报/审批推进时异步回写（`.catch(() => {})` 失败不阻塞主流程）。

## 四、链路追踪与可调试性（考点 5）

- V3 每次调用生成 **Request ID**（`v3-<时间戳36>-<nanoid6>`），随 `x-request-id` 头下发；
  V2 在所有响应头回传 `x-request-id`。
- V3 侧每次调用**无论成败**写入 `sync_log` 表，字段含：
  `request_id, api_name, method, params_summary, status_code, success, duration_ms, error_code, error_message, created_at`。
- **错误区分**（不统一抛 Internal Server Error）：
  | 场景 | error_code | status_code |
  |---|---|---|
  | 运单不存在 | `WAYBILL_NOT_FOUND` | 404 |
  | 鉴权失败 | `INVALID_CREDENTIALS` | 401 |
  | 网络超时 | `TIMEOUT` | null |
  | 网络错误 | `NETWORK_ERROR` | null |
  | V2 内部错误 | `INTERNAL_ERROR` / `HTTP_5xx` | 5xx |
- 通过 Request ID 可在 `/monitor` 页面还原一次完整调用链（时间、接口、入参摘要、状态码、耗时、错误信息）。

## 五、超时与重试策略

- **超时**：默认 8 秒（`V2_TIMEOUT_MS`），`AbortController` 控制。超时向用户明确提示
  「调用 V2 超时」，前端 15 秒兜底超时，**不无限转圈**。
- **重试**：幂等 **GET** 接口失败自动重试最多 **2 次**（指数退避 200ms×attempt）；
  4xx（除 429）不重试。**写接口（POST flag）不自动重试**，避免重复副作用。
- **幂等性保证**：GET 天然幂等；`sku-check` 虽为 POST 但语义只读、无副作用，重试安全；
  `flag` 回写以 `external_code` 为主键 UPSERT，重复写结果一致（幂等），故即便手动重试也安全。

## 六、V2 不可用时的降级方案

- 关键动作（上报/扫描）**必须实时校验**：
  - V2 明确返回 404 → 判定运单不存在，拒绝上报（真实性校验失败）。
  - V2 超时/网络错误/5xx → 返回 **503 `V2_UNAVAILABLE`**，前端明确提示「V2 不可用，请稍后重试」，
    **不允许**在无法校验真实性时创建工单（避免脏数据）。
- 只读展示（工单详情的运单信息）**可降级**：
  - V2 挂了时回退本地快照，前端明确标注 **「本地快照，同步于 XX 时间」**，
    不让用户误以为等于 V2 最新状态。
- V2 恢复后自动继续正常工作，无需人工介入（每次调用独立尝试实时接口）。

## 七、老系统二开意识：在存量 V2 上新增对外接口（考点 5）

我在 V2 上新增接口时遵循以下工程原则，**不破坏 V2 现有调用方**：

1. **命名空间隔离 + 版本化**：新接口全部置于 `/api/v3/*`，与 V2 既有的 `/api/orders`、
   `/api/import` 等完全隔离。未来接口演进走 `/api/v3.1/*` 或在响应体加 `apiVersion` 字段，
   老版本保留一段弃用期（sunset header 提示），灰度迁移。

2. **只增不改**：仅**新增** DB 方法（`getOrderByCode`、`get/setExceptionFlag`）与新表
   （`waybill_exception_flags`），未改动 V2 既有表结构与既有接口的入参出参。V2 原有功能零回归。

3. **字段向后兼容原则**：
   - 新增字段一律**可选 + 有默认值**，老调用方不传不受影响。
   - 不删除、不重命名、不改变既有字段类型。
   - 响应体只增字段不减字段（老客户端忽略未知字段即可）。

4. **灰度上线方案**：新接口通过独立环境变量 `V3_API_KEY` 控制访问，未配置 key 时可用开发默认值
   跑联调；生产灰度时先小流量验证，再全量。新接口异常不影响 V2 主链路（导入/解析）。

### V2 字段类型升级如何应对（题目示例：金额 int → decimal）

- **感知**：接口响应中约定携带 `apiVersion` / schema 版本；V3 侧对关键字段做**类型宽容解析**
  （`Number(x)` 兼容 int 与 decimal 字符串），不假设精确类型。
- **应对**：
  - V3 读取金额时统一走 `Number()` 归一，int→decimal 升级对 V3 透明。
  - 若涉及精度（decimal 小数位），V3 用 `Number` 存 REAL 已足够展示；如需严格对账，
    可将金额以字符串透传并在赔付对账环节用 decimal 库处理。
  - 通过 `sync_log` 的响应快照可回溯字段变化，必要时对账告警。
- **原则**：V3 对 V2 字段**做防御性解析**，不硬编码类型假设，是存量系统对接最稳妥的姿势。
